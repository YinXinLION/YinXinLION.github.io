排序检索数据

子句： SQL语句由子句构成，有些子句是必须的，而有的是可选的。一个子句通常由一个关键字和所提供的数据组成。

为了明确地排序用SELECT语句检索出的数据，可以使用ORDER BY子句。ORDER BY子句取一个或多个列的名字，据此对输出进行排序。
SELECT prod_name FROM products ORDER BY prod_name;
指示MYSQL对prod_name列以字母顺序排序数据。

通过非选择列进行排序也是合法的。

多个列排序： 为了按多个列排序，只要指定列名，列名之间用逗号分开即可。

先按价格排序，然后再按名称排序。

指定排序方向： 数据排序不限于升序排序(从A到Z)。这只是默认的排序顺序，可以使用ORDER BY子句降序(从Z到A)顺序排序。为了进行降序排序，必须指定DESC关键字。

prod_price列以降序排序，而prod_name列仍按标准的升序排序。

在多个列上降序排序 如果想在多个列上进行降序排序，必须对每个列指定DESC关键字。

在字典排序中，A被视为与a相同，这是MySQL的默认行为。

ORDER BY 子句的位置 在给出 ORDER BY 子句时，应该保证它位于 FROM 子句之后。如果使用LIMIT ，它必须位于 ORDER BY之后。使用子句的次序不对将产生错误消息。

MySQL必知必会学习笔记(2)

使用MySQL

选择数据库
在最初连接到MYSQL时，没有任何数据库打开供你使用。在执行任意数据库时，需要选择一个数据库。可使用关键字USE。

一定要使用use打开数据库，才能读取其中的数据。

了解数据库和表
数据库、表、列、用户、权限等的信息被存储在数据库和表中。内部的表一般不直接访问。可通过SHOW命令来显示这些信息
show databases;返回可用数据库的一个列表。

show tables;返回一个数据库内的表的列表

show columns from tablename对每个字段返回一行，行中包括字段名、数据类型、是否允许NULL、键信息、默认值以及其他信息(如字段cust_id的auto_increment)

自动增量：在每个行添加到表中时，MySQL可以自动地为每个行分配下一个可用编号，不用在添加一行时手动分配唯一值

DESCRIBE语句：支持用DESRIBE作为SHOW COLUMNS FROM的一张快捷方式。

show status; 用于显示广泛的服务器状态信息
show create database和show create table显示创建特定数据库或表的MySQL语句
show grants;用于显示授予用户的安全权限
show errors和show warnings用来显示服务器错误或警告信息

检索数据

检索单个列： SELECT prod_name FROM products; 从products表中检索一个名为prod_name的列。
SELECT后的结果是未排序的数据，可能查询出来顺序不一样。

SQL大小写：关键字使用大写，所有列和表名使用小写。

检索多个列： SELECT prod_id, prod_name, prod_price FROM products; 从products中选择数据。指定了3个列，列名之间用都好分隔。

检索所有列： SLELCT * FROM products; 如果给定一个通配符*，则返回表中所有列。

注意：不用明确列出所需列，但检索不需要的列通常会降低检索和应用程序的性能。

关键字DISTINCT,指示MySQL只返回不同的值。如果使用DISTINCT关键字，它必须直接放在列名的前面。应用于所有列而不仅是前置它的列。
不能部分使用 DISTINCT，SELECT DISTINCT vend_id, prod_price，除非指定的两个列都不同，否则所有行都将被检索出来。

限制结果： SELECT prod_name FROM products LIMIT 5; LIMIT 5 指示MySQL返回不多于5行。
LIMIT 5, 5 指示MySQL返回从行5开始的5行。第一个数为开始位置，第二个数为要检索的行数。

MySQL必知必会学习笔记(1)

了解SQL

数据库：保存有组织的数据的容器(通常是一个文件或一组文件)。

表：某种特定类型数据的结构化清单。(数据库中的每个表都有一个名字，用来标识自己。表名唯一)
表具有一些特性，这些特性定义了数据在表中如何存储，如可以存储什么样的数据，数据如何分解，各部分信息如何命名。
不同数据库可以有相同的表名。

列：表中的一个字段。所有表都是由一个或多个列组成的。(可以想象表是一个网格，网格中的每一列都存储着一条特定的信息。例如：顾客表，一列存储顾客编号，另一列存储顾客姓名，还有列存储地址)

正确将数据分解为多个列极为重要。这样可以对数据进行排序/过滤。

数据类型：所容许的数据的类型。每个表列都有相应的数据类型，它限制(或容许)该列中存储的数据。

行：表中的一个记录。(表中所保存的每个记录存储在自己的行内)。

主键：一列(或一组列)，其值能够唯一区分表中每个行。
没有主键，更新或删除表中特定行会很困难，因为没有安全的方法保证只涉及相关的行。

作为主键需要满足的条件：1.任意两行都不具有相同的主键值 2.每个行都必须具有一个主键值（主键列不允许NULL值）。

主键需要注意的地方：

  1. 不更新主键列中的值
  2. 不重用主键列的值

SQL：结构化查询语言(Structured Query Language). 与数据库打交道。

MySQL数据库是基于客户机-服务器的数据库。服务器负责所有数据访问和处理的一个软件。数据更新、添加、删除都由服务器软件完成(允许在数据库服务器上)。客户机上运行着发起请求的软件。

服务器软件：MYSQL DBMS。
客户机：MYSQL提供的工具，或开发语言。

例如：如果你请求一个按字母顺序列出的产品表，则客户机软件通过网络提交该请求给服务器软件。服务器软件处理这个请求，根据需要过滤、丢弃和排序数据；然后把结果送回到你的客户机软件。

mysql命令行程序(MySQL连接)

mysql -u ben -p -h myserver -P 9999用户名、主机名、端口和口令
注意：

  1. 命令输入在mysql>之后。
  2. 命令用;或\g结束，仅按Enter不执行命令。
  3. 输入help或\h获得帮助。(输入help select获得使用select语句的帮助)

设计模式

代理模式

代理模式：为另一个对象提供一个替身或占位符以控制对这个对象的访问。

让我们来看一下远程代理：

本地堆(客户对象->客户辅助对象) | 远程堆(服务辅助对象->服务对象)
客户对象：以为在和正真的服务沟通,以为客户辅助对象就是能够真正做事情的东西。
客户辅助对象：这是个代理对象。他假装自己是服务。
服务辅助对象：从客户辅助对象处取得请求,对它解包,调用真正服务上的方法。
服务对象：真正提供服务的方法,他的方法真正在做事情。

java RMI提供了客户辅助对象和服务辅助对象，为客户辅助对象创建和服务对象相同的方法。
RMI：将客户辅助对象成为stub(桩)，服务辅助对象成为skeleton(骨架).

客户并不知道如何去跟远程对象沟通，代理控制了远程对象的访问，客户去操作代理对象。
被代理的对象可以是：远程对象(远程代理)、创建开销大的对象(虚拟代理)、或需要安全控制的对象(保护代理)。

代理模式和装饰者模式很像：但是代理模式是为其他对象提供一种代理以控制对这个对象的访问。装饰者模式是用来增强增加对象行为的。他们结构类似，但目的不同哦。

动态代理：运行时才将它的类创建传来，所以叫动态代理。可以通过方法isProxyClass()来判断这个类是否是代理类。
通过Proxy.newProxyInstance()方法动态地创建代理。

防火墙代理：控制网络资源的访问，保护主题免于”坏客户”的侵害。
智能引用代理：当主题被引用时，进行额外的动作，例如计算一个对象被引用的次数。
缓存代理：为开销大的运算结果提供暂时存储，它也允许多个客户共享结果，以减少计算或网络延迟。
同步代理：在多线程的情况下为主题提供安全的访问。
复杂隐藏代理：也成为外观模式，代理控制访问。
写入时复制代理：用来控制对象的复制，方法是延迟对象的复制，直到客户真的需要为止。

远程代理管理客户和远程对象之间的交互。
虚拟代理控制访问实例化开销大的对象，可以在访问时做一些处理(启动一个线程进行创建对象，防止因等待对象而挂起)、
保护代理基于调用者控制对象方法的访问。做一些处理(判断这个方法名是否允许执行)

代理会造成你的设计中类的数目增加。

设计模式

状态模式

状态模式，它通过改变对象内部的状态来帮助对象控制自己。

状态转换机：当执行某一步时，会从上一步的状态转换成下一步的状态。
例如投币机：一开始状态为(没有币)，当你投币了变成(有币)状态，当你摇动曲柄了变成(出货)状态，当出货后变成(没有币)状态。
简单实现

加入，突然又要加一种状态进去，例如曲柄摇动后，可能会有另外奖励状态，如果使用状态转换机，那么我们要修改原有代码，并保证他不会出错。

我们需要不维护我们现有的代码，我们重写它以便于将状态对象封装在各自的类中，然后再动作发生时委托给当前的状态。(一个状态是一个对象，委托给状态执行需要的任务)。

通过从结构上改变实现，做到了一下几点：

  1. 将每个状态的行为局部化到它自己的类中。
  2. 将容易产生问题的if语句删除，以便日后维护。
  3. 让每一个状态对”修改关闭”，对”扩展开放”(可以添加新的状态类呀)。

状态模式 :允许对象在内部状态改变时改变它的行为，对象看起来好像修改了它的类。

要点：

  1. 和程序状态机不同，状态模式用类代表状态。
  2. 状态模式和策略模式有相同的类图，但是他们的意图不同。(策略：由子类决定如何实现算法中的某些步骤 状态：封装基于状态的行为，并将行为委托到当前状态)
  3. 状态可以由State类或Context(machine类)控制。
  4. 状态类可以被多个Context实例共享。

理解同步、异步、阻塞、非阻塞(美团新零售2面) 2018-4-11

首先，通过美团的面试让我认识到我的不足之处，除了学外还要多思考多总结。
同步、异步关注的是 消息通信机制。
同步：发出一个调用，在调用没有得到结果前，调用就不返回。一旦调用返回，就能得到返回值。
异步：发出一个调用后，调用直接返回，所以没有结果。被调用者通过状态、通知来通知调用者，或者通过回调函数处理这个调用。

阻塞、非阻塞关注的是 程序在等待调用结果(消息、返回值)时的状态
阻塞：调用结果返回之前，线程会被挂起，线程得到结果之后才会返回。
非阻塞：不能立刻得到结果之前，该调用不会阻塞当前线程。

工厂方法模式

定义了一个创建对象的接口，但由子类决定创建的对象是哪一个。工厂方法让类把实例化推迟到子类。
抽象工厂：提供一个接口，用于创建相关或依赖对象的家族，而不需要明确指定具体类。定义一个负责创建一组相关的产品的接口。

命令模式 2018-4-12

命令模式 将”请求”封装成对象(命令对象),以便使用不同的请求、队列或者日志来参数化其他对象。命令模式也支持可撤销的操作。
形象生动的例子：Lion去餐馆点餐，通过createOrder()创建一个订单，这个订单通过takeOrder()函数交给服务员，然后调用orderUp()方法通知厨师准备餐点，然后厨师执行makeBurger(),makeShake()等方法，output出甜点和汉堡。

Lion是Client，订单是Command，服务员是调用者，通过takeOrder()就是将命令对象(订单)交给调用者，然后orderUp相当于让调用者执行execute方法，告诉接收者(厨师),execute里执行的就是接收者工作的细节。 客户与接收者实现解耦。

适配器模式和外观模式 2018-4-12

适配器模式： 将一个类的接口，转换成客户期望的另一个接口。适配器让原本接口不兼容的类可以合作无间。
分为对象适配器和类适配器（需要多重继承，java不支持）

客户使用适配器的过程：1.客户通过目标接口调用适配器的方法对适配器发出请求 2.适配器使用被适配者接口把请求转换成被适配者的一个或多个调用接口。 3.客户接收到调用的结果，但并未察觉这一切是适配器在起转换作用。

装饰者模式是用来增强行为，适配器模式是起到接口转换作用，如何去调用被适配者的。

外观模式：：提供一个统一的接口，用来访问子系统中的一群接口。外观定义了一个高层接口，让子系统更容易使用。外观模式的意图是简化接口。

“最少知识原则”:不要让太多的类耦合在一起。(让我们的朋友圈子维持在最小的状态)。

模板方法模式

模板方法模式：在一个方法中定义一个算法的骨架，而将一些步骤延迟到子类中。模板方法使得子类可以在不改变算法结构的情况下，重新定义算法中的某些步骤。

“好莱坞原则”：高层组件会决定如何调用底层组件，不要让底层组件去调用高层。

在JAVA api中可以看到Array.sort()使用了模板方法模式。

观察Comparable对象的compareTo,这个就是子类要做的事情，要重新去实现，子类实现Comparable接口，并且实现compareTo方法。 这只是api其中一个运用到模板方法模式。还有java.io的InputStream的read()方法，这个方法又会被read(byte b[], int off, int len)模板方法使用。

策略模式和模板方法模式都封装算法，一个用的是 组合，一个用的是 继承。
模板方法模式：让代码重用，部分实行延迟到子类(工厂方法是模板方法一个特殊版本)。

迭代器模式 2018-4-13

迭代器模式： 提供一种方法顺序访问一个聚合对象中的各个元素，而又不暴露其内部的表示。客户端不需要知道聚合对象内部是如何实现的，只需要使用统一的接口即可。

迭代器模式让我们能游走于聚合内的每一个元素，而又不暴露其内部的表示。有一个统一的方法访问聚合中的每一个对象。让聚合专注应该的事情上，让接口与实现变得更简洁。

“单一责任”：一个类应该只有一个引起变化的原因。
当一个模块或一个类被设计成只支持一组相关的功能时，我们说它具有高内聚;反之，当被设计成支持一组不想关的功能时，我们说它具有低内聚。

组合模式 2018-4-13

组合模式：允许你将对象组合成树形结构来表现”整体/部分”层次结构。组合能让客户以一致的方式处理个别以及对象组合。
使用组合结构，我们能够把相同的操作应用在组合和个别对象上。(大多情况下，我们可以忽略对象组合和个别对象之间的差别。)
组合模式让我们能用树形方式创建对象的结构，树里面包含了组合以及个别的对象。

例如：最顶层的菜单，拥有所有的菜单和菜单项，顶层菜单下面有A菜单，B菜单，C菜单。C菜单下面有很多菜单项。
这些菜单和菜单项都继承自同一个接口，但是有些菜单的功能，菜单项却不适合调用，这就要求客户端编写时保证不会出现错误情况发生，捕获异常方式解决。

组合模式：客户可以将对象的集合以及个别的对象一视同仁。(用一致的方式对待所有对象)

Zookeeper与Paxos

Zookeeper介绍

Zookeeper为分布式应用提供了高效且可靠的分布式协调服务，提供了诸如统一命名服务、配置管理和分布式锁等分布式的基础服务。在解决分布式一致性方面，采用了ZAB(Zookeeper Atomic Boradcast)的一致性协议。

Zookeeper是什么

Zookeeper是一个典型的分布式数据一致性的解决方案。分布式应用程序可以基于它实现诸如数据发布/订阅、负载均衡、命名服务、分布式协调/通知、集群管理、Master选举、分布式锁、分布式队列等功能。

Zookeeper保证如下分布式一致性特性：
1.顺序一致性：客户端发起事务请求，严格按照发起顺序被应用到Zookeeper中。
2.原子性：事务请求处理结果在整个集群中所有机器上的应用情况是一致的。要么都成功应用了某一个事务，要么都没应用。
3.单一视图：无论客户端连接哪个Zookeeper服务器，看到服务端数据模型都是一致的。
4.可靠性：服务端成功应用了一个事务，那么状态会一直保留，直到其他事物对其进行变更。
5.实时性：最终一致性，Zookeeper仅仅保证在一定时间段内，客户端最终一定能够从服务端上读取到最新的数据状态。

Zookeeper设计目标

1.简单的数据模型:ZooKeeper数据模型是树型结构的名字空间。
2.可以构建集群：只要集群中存在超过一半的机器能够正常工作，那么就能够正常对外服务。当客户端和Zookeeper服务器断开后，客户端会自动连接到集群中的其他机器。
3.顺序访问：对于每一个更新请求，都会分配一个全局唯一的递增编号，保证事务操作先后顺序。
4.高性能：Zookeeper将全量的数据存储在内存中，以此获得服务器吞吐量、减少延迟。

Zookeeper的基本概念

集群角色

引入Leader、Follower和Observer三种角色。Leader服务器为客户端提供读和写服务。Follower和Observer能够提供读服务。
唯一区别在于：Observer不参与Leader选举过程，不参与写操作的”过半写成功策略”。

会话

Session的sessionTimeout是设置一个客户端会话的超时时间。当由于各种原因导致客户端断开连接，只要在sessionTimeout规定时间内重新连上集群任意一台服务器，那么创建的会话仍然有效。

数据节点(Znode)

Znode是数据模型中的数据单元，Zookeeper将所有数据都存储在内存，数据模型是一颗树(Znode Tree),由斜杠(/)进行分割的路径就是一个Znode，例如/foo/path1。Znode除了保存自己的数据内容，还保存一系列属性信息。
Znode分为持久节点和临时节点。持久节点：Znode被创建后，除非主动进行Znode移除操作，否则Znode一直保存在Zookeeper上。临时节点：它的生命周期和客户端会话绑定，一旦客户端会话失效，那么这个客户端创建的所有临时节点都会被移除。
特殊属性：SEQUENTIAL。是父节点维护的自增数字。

版本

每个Znode维护一个Stat的数据结构，Stat记录Znode的三个数据版本，version(当前ZNode的版本)、cversion(当前Znode子节点的版本)和aversion(当前ZNode的ACL版本)。

Wathcer

Zookeeper学习-一致性协议

一致性协议

二阶段提交


1.提交事务请求 2.执行事务提交
如果任何一个参与者返回No，或者等待超时时间后，协调者无法接收所有的反馈，那么执行事务中断。

优点：原理简单、实现方便。
缺点：
1.同步阻塞：各个参与者在等待其他参与者响应的过程中，将无法进行其他任何操作。
2.单点问题：一旦协调者出现问题，那么整个二阶段提交流程将无法运转。参与者会一直处于锁定事务资源的状态中。
3.数据不一致性：由于网络的原因或者协调者自身宕机，导致只有部门参与者收到Commit请求，于是整个分布式出现数据不一致现象。
4.太过保守：任意一点失败都会导致事务的失败。

二阶段提交协议解决了分布式事务的原子性问题

三阶段提交


1.选票收集CanCommit 2.提交授权PreCommit 3.确定提交DoCommit

优点：降低参与者的阻塞范围，能够在出现单点故障后继续达成一致。
缺点：参与者接收到PreCommit消息后，如果网络出现分区，协调者所在的节点和参与者无法正常通信，参与者依然会进行事务提交，导致数据不一致。

三阶段提交避免了二阶段提交中无限等待问题。

Paxos算法


然后通知Leaner集合的一部分，然后通知剩余Leaner。保证可靠性，防止通知1个Leaner导致单点故障。

P1：提出编号为M1的提案，完成阶段1.
P2:提出编号为M2的提案（M2>M1），完成阶段1.此时不批准小于M2的提案。
P1：进入阶段二，发出的Accept请求会被忽略。于是P1重新生成编号M3的提案(M3>M2),完成阶段1。
然后无限循环。为了避免这一现象，必须采用一个主Proposer。这样整套Paxos算法流程能够保持活性。

Paxos算法解决了无限等待问题，解决了”脑裂”问题。

Zookeeper学习笔记(1)

分布式架构

从集中式到分布式

由于集中式系统规模大，运维人员难以维护，加上容易出现单点故障，PC机性能的不断提升，导致慢慢走向分布式系统时代。

集中式特点

集中式系统就是指由一台或多台主计算机组成中心节点，数据集中存储于这个中心节点，所有业务单元也都部署在中心节点上。
特点：部署结构简单。需要有性能卓越的主机。

分布式特点

定义：通过消息传递进行通信和协调工作的。
特征：1.分布性 2.对等性 3.并发性 4.缺乏全局时钟 5.故障总是会发生。

分布式环境的会遇到的问题

1.通信异常：引入网络因素，网络本身是不可靠的，可能导致分布式无法顺利完成一次网络通信。
2.网络分区：分布式系统中部分节点之间的网络延时不断增大，最终导致分布式系统中只有部分节点能够正常通信，另一些节点不能。我们将这个现象称为网络分区，或者叫”脑裂”。
3.三态：网络会出现各种各样的问题，每一次请求或响应，可能会存在成功、失败、超时。
4.节点故障：每个节点都会出现故障，服务器节点可能会出现宕机或”僵死”。

ACID到CAP/BASE

ACID

事务：由一系列对系统中数据进行访问与更新的操作所组成的一个程序执行逻辑单元。
事务的4个特征：1.原子性 2.一致性 3.隔离性(读取未提交、读取已提交、可重复度、序列化) 4.持久性。

分布式事务

分布式事务：是指事务的参与者、支持事务的服务器、资源服务器以及事务管理器分别位于分布式系统的不同节点之上。通常一个分布式事务中会涉及对多个数据源或业务系统的操作。

CAP和BASE

CAP定理：一个分布式系统不可能同时满足一致性、可用性、分区容错性。最多只能满足其中两项。
一致性：多个副本之间是否能够保持一致的特性。
可用性：指系统提供的服务必须一致处于高可用状态，对于用户的每一个操作请求总是能够在有限的时间内返回结果。系统必须存在一个合理的响应时间，且返回一个正常的响应结果（要么成功，要么失败）。
分区容错性：故障发生时，仍能够保证对外提供满足一致性和可用性的服务，除非整个网络环境都发生了故障。

BASE定理：即使无法做到强一致性，但每个应用都可以根据自身的业务特点，采用适当的方式来使系统达到最终一致性。

最终一致性的五类主要变种：
1.因果一致性：A更新完数据后通知B，B能够访问A的最新值，如果B要进行更新，必须基于A更新后的值。同时，与A无因果关系的C没有此限制。
2.读己之所写：A更新数据后，自己总能访问到更新过得最新值，不会看到旧值。
3.会话一致性：执行更新操作后，客户端能够在统一个会话中始终读取到该数据项的最新值。
4.单调读一致性：如果一个进程从系统中读取出一个数据项的某个值后，那么系统对于该进程后续的任何数据访问都不应该返回更旧的值。
5.单调写一致性：一个系统需要能够保证来自同一个进程的写操作被顺序的执行。
